What is the problem?

Managing ever-changing state is hard

  • One change could lead to several others
  • Complex state changes are hard to reason about
  • Bugs may be hard to reproduce
Many Components accessing the same state and Changing it Imagine people working on docs in same folder. Hard to know who changed what

Shared mutable state is evil

It gets hard to trace errors if there are many dependencies between components. Hard to trace who made an error

Redux

A predictable state container for JavaScript applications

Try to explain with a real world analogy. Address also none programmers

Core Principles

  • Single Source of Truth
  • State is read only
  • Changes are made with pure functions (called Reducers)
  • Single Source of Truth => Sort of "Library" where all information is kept.
  • Read Only => can't go directly to the library and change sth
  • Pure Functions => Workers can only do specific things, know how to handle actions/instructions

Store

The store is a single JavaScript Object which represents the state tree

The store is our "Library" where all the necessary information is stored

The store is our Single Source of Truth

Every state change goes through it

Actions

Actions specify that something happened

Actions are like instructions for workers in our "Library"

                    
                    {
                        type: "ADD_STH_COOL"
                        payload: "My extra cool stuff"
                    }
Actions are just plain Javascript objects. 'type' property is mandatory! 
                    
                    {
                        type: "ADD_AWESOME_STUFF"
                        payload: "My extra cool stuff"
                    }
for visualization purposes

=>

Reducers

Reducers compute new state based on actions

Reducers are the workers/staff in our "Library"

Only part of the app that can "change" state

Reducers need to know

  • The current state
  • The action to carry out
                    
                        (state, action) => state

They don't mutate the state. They create a new one!

reducers are like paranoid workers. They make a copy before they change anything. You would think: this is not performant. But it is. Structural sharing. Won't go into details (questions later)

Why Redux?

  • Easy to reason about data flow
  • Errors/bugs are easy to trace
  • State is reproducable by "replaying" actions
  • Performance benefits
  • Cool developer tools
    • Time Travel Debugging
    • Hot Reloading
